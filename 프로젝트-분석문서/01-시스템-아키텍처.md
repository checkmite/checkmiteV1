# 01. 시스템 아키텍처

## 1. 전체 구조 (3-tier + AI 런타임)

CheckMite는 4개의 독립 프로세스로 구성되며, 운영 시 단일 서버 안에 함께 배치합니다.

```
                         외부 브라우저
                              │
                              ▼
                    ┌──────────────────┐
                    │   nginx (80/443) │   정적 파일 서빙 + /api 프록시
                    └──────────────────┘
                       │              │
            정적 파일   │              │  /api/*
         ┌─────────────┘              └──────────────┐
         ▼                                            ▼
 ┌────────────────┐                         ┌──────────────────────┐
 │  React 빌드물   │                         │  Express Backend :3000 │
 │  (dist/)       │                         │  - REST API            │
 └────────────────┘                         │  - 파일 업로드(multer)  │
                                            │  - 비즈니스 로직         │
                                            └──────────────────────┘
                                               │                 │
                                  파일경로/업로드 │                 │ SQL
                                               ▼                 ▼
                                  ┌────────────────────┐  ┌──────────────┐
                                  │ FastAPI Model :8000 │  │ PostgreSQL    │
                                  │ - YOLOv8 추론       │  │ :5432         │
                                  │ - OpenCV 영상처리   │  └──────────────┘
                                  │ - 추적/활력도 계산  │
                                  └────────────────────┘
```

> **보안 원칙**: 외부에는 nginx(80/443)만 노출. Express·모델서버·PostgreSQL은 내부망(127.0.0.1)에서만 접근.

## 2. 컴포넌트별 책임

### Frontend (React + Vite)
- 5개 탭 UI 렌더링 (탐지/밀도/증식률/박스관리/휴지통)
- 결과 시각화 (바운딩박스 오버레이, SVG 차트, 트래킹 영상 재생)
- 파일 업로드 + 진행률 표시
- **라우터 없음**: `tab` state로 조건부 렌더링

### Backend (Express)
- REST API 엔드포인트 제공
- 파일 업로드 수신·저장 (`multer`)
- 사육박스 CRUD, soft delete/복구
- 분석 작업 오케스트레이션 (모델서버 호출 → 결과 가공 → DB 저장)
- 증식률 계산 (DB 이력 집계)
- 빌드된 프론트엔드 정적 서빙 (옵션)
- Swagger UI 문서 (`/api/docs`)

### Model Server (FastAPI)
- YOLOv8m 가중치 로드 (필요 시 분할 파일 자동 병합)
- 이미지 타일 추론 (객체 탐지)
- 영상 프레임 샘플링 + 화질 평가 (밀도)
- 객체 추적 + 속도 기반 활력 점수 산출 (활력도)

### Database (PostgreSQL)
- 사육박스, 업로드 파일, 분석 작업, 측정 결과, 상세 결과, 휴지통 이력을 **정규화 분리** 저장

## 3. 통신 방식

| 구간 | 프로토콜 | 데이터 형식 | 비고 |
|------|----------|------------|------|
| 브라우저 → Express | HTTP | JSON / multipart | `/api` prefix |
| Express → 모델서버 (detection) | HTTP POST | **multipart (파일 바이너리)** | `/predict/image` |
| Express → 모델서버 (density/vitality) | HTTP POST | **JSON (파일 경로 전달)** | `/infer/density`, `/infer/vitality` |
| Express → PostgreSQL | TCP | SQL (pg 드라이버) | 커넥션 풀 |

> **핵심 설계**: detection은 이미지를 바이너리로 전송하지만, density/vitality는 무거운 영상을 재전송하지 않고 **로컬 파일 경로만 JSON으로 전달**한다. 같은 서버에 있기에 가능한 최적화.

## 4. 대표 데이터 흐름

### (A) 객체 탐지 (동기)
```
사용자 이미지 업로드
  → POST /api/analysis/detection (multipart)
  → analysis.service.runDetection()
  → modelRuntimeService.inferDetection() → POST :8000/predict/image
  → YOLOv8 타일 추론 → 박스 좌표 정규화
  → DB 저장 (measurements + detection_boxes)
  → 응답 반환 → 프론트 바운딩박스 렌더링
```

### (B) 밀도+활력도 통합 분석 (비동기 + 폴링)
```
사용자 영상 N개 업로드 (클라이언트 10초 검증)
  → POST /api/analysis/density → 즉시 jobId 반환
  → setImmediate 백그라운드 시작
       각 영상마다:
         1) inferDensity → :8000/infer/density (프레임 샘플링/화질평가/카운트)
         2) inferVitality → :8000/infer/vitality (추적/속도/활력점수)
  → 프론트는 1초마다 GET /api/analysis/density/:jobId/progress 폴링
  → 완료 시 DB 저장 (density/vitality measurement 각각)
  → 최종 결과 반환 → 결과 화면 + 트래킹 영상
```

### (C) 증식률 분석 (추론 없음)
```
탭 진입
  → GET /api/culture-boxes/:boxId/growth  (집계 계산)
  → GET /api/culture-boxes/:boxId/measurements  (이력)
  → growth.service가 DB 이력으로 변화율/로그증식률/가중 증식률 계산
  → 프론트 SVG 차트 + 카드 렌더링
```

## 6. 계층 분리 원칙 (백엔드)

```
routes  →  controllers  →  services  →  repositories  →  PostgreSQL
 (URL)      (req/res)      (로직)        (SQL)

                            services  →  model-runtime.service  →  FastAPI
```

- **routes**: URL과 controller 연결만
- **controllers**: 요청 파싱 / 응답 직렬화만
- **services**: 사육박스 관리, 분석 오케스트레이션, 증식률 계산 등 핵심 로직
- **repositories**: SQL 쿼리 전담
- **model-runtime.service**: Express ↔ 모델서버 HTTP 통신 담당
