# 03. 백엔드 분석 (Express)

## 1. 기술 스택

| 항목 | 기술 | 비고 |
|------|------|------|
| 런타임 | Node.js (ESM `type: module`) | |
| 프레임워크 | Express 5.2 | |
| DB 드라이버 | `pg` 8 | PostgreSQL 커넥션 풀 |
| 파일 업로드 | `multer` 2 | 디스크 스토리지 |
| CORS | `cors` | |
| 환경변수 | `dotenv` | |
| API 문서 | `swagger-ui-express` | `/api/docs` |

---

## 2. 부트스트랩 흐름

```
server.js  →  createApp()  →  app.listen(port)
   │
   ├─ env.js          환경변수 로드 (.env)
   ├─ database.js     pg Pool 생성
   └─ SIGINT/SIGTERM  graceful shutdown (pool.end)
```

`server.js`는 앱 생성·리슨·종료 시그널 처리만 담당. 라우트/미들웨어 등록은 `app.js`.

---

## 3. 미들웨어 파이프라인 (`app.js`)

```js
cors()                                   // 모든 출처 허용
express.json({ limit: '5mb' })           // JSON 본문
express.urlencoded({ extended: true })   // 폼 본문

GET  /api/docs        → Swagger UI
GET  /api/health      → DB 핑(SELECT 1) 확인
GET  /uploads/*       → 업로드 파일 정적 서빙 (트래킹 영상 등)

/api/culture-boxes    → cultureBoxRoutes
/api/analysis         → analysisRoutes
/api/measurements     → measurementRoutes
/api/trash            → trashRoutes

(옵션) FRONTEND_DIST_DIR 존재 시 → 정적 프론트 + SPA fallback
notFoundMiddleware → errorMiddleware
```

> `FRONTEND_DIST_DIR`이 있으면 **단일 Express가 프론트까지 서빙**. `/api/*`가 아닌 모든 경로는 `index.html`로 폴백(SPA 라우팅).

## 4. 계층 구조

```
routes → controllers → services → repositories → PostgreSQL
                          ↓
              model-runtime.service → FastAPI(:8000)
```

### 4.1 환경설정 (`config/env.js`)
| 변수 | 기본값 | 의미 |
|------|--------|------|
| `PORT` | 3000 | 백엔드 포트 |
| `DATABASE_URL` | `postgresql://checkmite:password@127.0.0.1:5432/checkmite` | DB 접속 |
| `UPLOAD_DIR` | `uploads` | 업로드 저장 경로 |
| `UPLOAD_MAX_IMAGE_MB` | 20 | 이미지 최대 크기 |
| `UPLOAD_MAX_VIDEO_MB` | 500 | 영상 최대 크기 |
| `MODEL_RUNTIME_URL` | `http://127.0.0.1:8000` | 모델서버 주소 |
| `MODEL_RUNTIME_REQUIRED` | `false` | true면 모델 실패 시 에러, false면 fallback |
| `FRONTEND_DIST_DIR` | `dist` | 정적 프론트 경로 |

### 4.2 업로드 미들웨어 (`middleware/upload.middleware.js`)
- `multer.diskStorage`로 디스크 저장
- 파일명을 **`randomUUID() + 확장자`**로 변경 (충돌·추측 방지)
- 용량 제한: 영상 기준 500MB
- 서버 시작 시 업로드 디렉터리 자동 생성

## 5. 핵심 서비스 로직

### 5.1 분석 서비스 (`services/analysis.service.js`)

세 가지 분석을 오케스트레이션하는 가장 두꺼운 서비스.

#### (A) `runDetection` — 객체 탐지 (동기)
```
모델서버 inferDetection 호출 → 박스 좌표 수신
→ 트랜잭션: uploaded_file + analysis_job + measurement + detection_boxes 저장
→ job 상태 completed → 결과 반환
```

#### (B) `startDensity` / `runDensity` — 밀도+활력도 (비동기)
```
startDensity:
  진행률 객체 생성 → setImmediate로 백그라운드 runDensity 실행 → 즉시 jobId 반환

runDensity (영상마다 2단계):
  1) inferDensity  → 프레임 카운트/화질평가
  2) inferVitality → 추적/속도/활력점수 (+1번 영상은 트래킹 영상 렌더)
  진행률 매 단계 갱신 (completedOps/totalOps)
  → 여러 영상 결과 집계 (평균 count, 최대 프레임, 평균 속도/활력)
  → 속도비 기반 경고 산출 (vitalityNoticeForSpeedRatio)
  → 트랜잭션: density measurement + vitality measurement 동시 저장
```

**집계 로직 핵심**:
- `estimatedCountPerMl` = 영상별 추정치 **평균**
- `bestFrameCount` = 영상별 최대 프레임 카운트 중 **최댓값**
- `densityPerLiter` = mL당 개체수 × 1000
- 대표 track 선정: live motion → vitality score → frames → 이동거리 순 정렬
- **속도비 경고**: ≤2x 위험 / ≤3x 주의 / 그 외 정상

#### (C) `runVitality` — 단독 활력도 (현재 UI 미사용)
밀도와 분리된 영상 1개 활력도 분석. API는 존재하나 프론트 탭에서 직접 호출 안 함.

### 5.2 진행률 서비스 (`services/analysis-progress.service.js`)
- **인메모리 `Map`**으로 작업 상태 관리 (DB 아님 → 서버 재시작 시 소실)
- `create/update/updateSample/complete/fail` 제공
- 샘플별 상태(`pending → density → vitality → completed/failed`)와 전체 percent 추적
- 프론트가 1초 폴링으로 조회

> ⚠️ 인메모리라 **수평 확장(다중 인스턴스) 시 공유 안 됨**. 단일 서버 전제 설계.

### 5.3 증식률 서비스 (`services/growth.service.js`)
DB 측정 이력만으로 계산 (추론 없음).
```
변화율        = (현재 - 최초) / 최초 × 100
로그증식률    = (ln(현재) - ln(최초)) / 일수
일 증가량     = (현재 - 최초) / 일수
통합 증식률   = count변화율 × 0.75 + 활력도변화율 × 0.25
```
- 첫 측정 대비 / 직전 측정 대비 두 관점 모두 계산
- 라벨: 직전 -20%↓ "주의 관찰" / 통합 +20%↑ "증식 활발" / -10%↓ "감소 추세" / 그 외 "유지 관찰"

### 5.4 사육박스 서비스 (`services/culture-box.service.js`)
- `delete`는 **soft delete + trash_event 기록**을 한 트랜잭션으로 처리
- 삭제 시 측정 이력을 trash_event payload에 스냅샷으로 보관

## 6. 모델 런타임 연동 (`services/model-runtime.service.js`)

Express ↔ FastAPI 통신 전담.

| 함수 | 방식 | 타임아웃 | fallback |
|------|------|---------|----------|
| `inferDetection` | multipart (파일 바이너리) → `/predict/image` | 120초 | ✅ 가짜 박스 반환 |
| `inferDensity` | JSON (파일경로) → `/infer/density` | 300초 | ❌ |
| `inferVitality` | JSON (파일경로) → `/infer/vitality` | 300초 | ❌ |

### fallback 메커니즘 (detection)
```js
catch (error) {
  if (env.modelRuntimeRequired) throw error;   // 운영: 에러 전파
  return { countValue: 12, boxes: [가짜 박스 2개] };  // 개발: 가짜 결과
}
```
모델 서버 없이도 프론트~백엔드 전체 흐름을 개발/시연할 수 있게 함.

### 좌표 정규화 (detection)
모델은 픽셀 좌표를 주지만 런타임 서비스가 **이미지 크기로 나눠 0~1 정규화** 후 프론트에 전달:
```js
x: d.box.x1 / imgW,  width: d.box.width / imgW, ...
```

## 7. 컨트롤러의 HTTP 상태 코드 규약

| 액션 | 코드 | 의미 |
|------|------|------|
| `density` (비동기 시작) | **202 Accepted** | 작업 접수, 폴링 필요 |
| `detection` / `vitality` (동기 완료) | **201 Created** | 측정 생성됨 |
| `densityProgress` | 200 | 진행 상태 조회 |

---

## 8. 백엔드 특징 정리

1. **깔끔한 계층 분리** — routes/controllers/services/repositories/schemas/middleware 역할 명확
2. **트랜잭션 일관성** — 분석 결과 저장, 삭제 등은 `withTransaction`으로 원자적 처리
3. **2가지 모델 통신 방식** — 이미지는 바이너리, 영상은 파일경로(같은 서버 최적화)
4. **인메모리 진행률** — 단순하지만 단일 서버 한정
5. **개발 친화 fallback** — 모델 서버 없이도 동작 가능
