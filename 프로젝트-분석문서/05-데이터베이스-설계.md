# 05. 데이터베이스 설계 (PostgreSQL)

## 1. 설계 원칙

> **하나의 테이블에 모든 정보를 몰아넣지 않는다.**

탭 이름에 맞춘 단일 테이블 대신, **사육박스 / 업로드 파일 / 분석 작업 / 측정 / 상세 결과 / 휴지통 이력**을 분리 저장.

### 분리 이유
- 사육박스 정보와 측정 결과는 변경 주기가 다름
- 밀도/활력도 결과는 조회·계산 방식이 다름
- 상세 모델 결과는 크기가 커서 별도 테이블/JSONB로 분리
- 삭제/복구 이력은 활성 데이터와 별도 추적
- 모델 결과 스키마가 바뀌어도 전체 테이블이 흔들리지 않게

---

## 2. 테이블 관계도 (ERD)

```
culture_boxes (사육박스)
   │ 1
   ├──< uploaded_files     (업로드 파일 메타)
   ├──< analysis_jobs      (분석 작업)
   ├──< measurements ──────┐ (측정 공통 레코드)
   │                       │ 1
   │                       ├──< density_results       (밀도 상세)
   │                       ├──< density_frame_counts  (프레임별 카운트)
   │                       ├──< vitality_results      (활력도 상세)
   │                       ├──< vitality_series       (활력도 시계열)
   │                       └──< detection_boxes       (탐지 박스)
   └──< growth_snapshots   (증식률 스냅샷, 선택적)

trash_events (삭제/복구 이력, 느슨한 참조)
```

---

## 3. 핵심 테이블

### culture_boxes — 사육박스
```sql
id UUID PK, name TEXT, started_at DATE, memo TEXT,
created_at, updated_at, deleted_at TIMESTAMPTZ
```
- **초기 개체수 저장 안 함** (설계 원칙)
- `deleted_at IS NULL` → 활성, 값 있으면 → 휴지통 (**soft delete**)

### uploaded_files — 업로드 파일
```sql
id, culture_box_id FK, original_name, mime_type,
file_size, storage_path, checksum, created_at
```
원본 파일명·MIME·크기·저장경로 메타데이터.

### analysis_jobs — 분석 작업
```sql
id, culture_box_id FK, uploaded_file_id FK,
type   ∈ (detection, density, vitality),
status ∈ (queued, processing, completed, failed),
error_message, created_at, started_at, completed_at
```
분석 요청 단위. 진행 상태 추적용.

### measurements — 측정 공통 레코드 (중심 테이블)
```sql
id, culture_box_id FK, analysis_job_id FK,
type ∈ (detection, density, vitality),
measured_at TIMESTAMPTZ,
count_value INTEGER,
density_per_cm2 NUMERIC,
density_per_liter NUMERIC,   -- 마이그레이션으로 추가
vitality_score NUMERIC,
active_ratio NUMERIC,
result_summary JSONB,        -- 모델 원본 결과 통째 보관
created_at
```
> 모든 분석의 공통 헤더. 상세는 각 결과 테이블로 분리, 원본 JSON은 `result_summary`에 보존.

## 4. 상세 결과 테이블

### density_results — 밀도 상세
```sql
measurement_id FK(CASCADE),
measured_area_cm2, peak_count, average_count, density_per_cm2,
best_frame_count, estimated_count_per_ml, density_per_liter,
count_multiplier, video_duration_seconds,
selected_frame_index, selected_frame_quality JSONB, density_grade
```

### density_frame_counts — 프레임별 카운트
```sql
measurement_id FK(CASCADE), frame_index, count_value, quality JSONB
```
영상 분석의 프레임 단위 원시 카운트.

### vitality_results — 활력도 상세
```sql
measurement_id FK(CASCADE),
vitality_score, active_ratio, average_speed_mm_per_sec
```

### vitality_series — 활력도 시계열
```sql
measurement_id FK(CASCADE), frame_index, score
```

### detection_boxes — 탐지 박스
```sql
measurement_id FK(CASCADE), class_name, confidence,
x, y, width, height    -- 정규화(0~1) 좌표
```

### growth_snapshots — 증식률 스냅샷 (선택적)
```sql
culture_box_id FK, from_date, to_date,
first/current_density_per_cm2, density_change_*,
density_growth_per_day, log_density_growth_per_day, result_json JSONB
```
> 증식률은 보통 **실시간 계산**(growth.service)하므로 필수는 아님. 결과 보관용.

### trash_events — 삭제/복구 이력
```sql
entity_type, entity_id, action ∈ (deleted, restored), payload JSONB
```
삭제 시 박스+측정 이력을 payload에 스냅샷으로 기록.

## 5. 인덱스

```sql
idx_culture_boxes_deleted_at          (deleted_at)        -- 활성/휴지통 필터
idx_measurements_box_measured_at      (culture_box_id, measured_at)  -- 추이 조회
idx_measurements_type                 (type)
idx_density_results_measurement       (measurement_id)
idx_vitality_results_measurement      (measurement_id)
idx_analysis_jobs_status              (status)
idx_trash_events_entity               (entity_type, entity_id)
```

## 6. 마이그레이션 전략

`schema.sql` 하단에 **멱등(idempotent) 진화 구문** 포함:
```sql
ALTER TABLE measurements ADD COLUMN IF NOT EXISTS density_per_liter ...;
ALTER TABLE density_results ALTER COLUMN measured_area_cm2 DROP NOT NULL;
ALTER TABLE density_results ADD COLUMN IF NOT EXISTS best_frame_count ...;
...
```
- `CREATE TABLE IF NOT EXISTS` + `ADD COLUMN IF NOT EXISTS`로 **반복 실행 안전**
- `npm run backend:migrate`(`db/migrate.js`)가 이 스키마를 적용
- 초기 설계(`density_per_cm2` 면적 기반) → 실제 구현(`density_per_liter` 영상 카운트 기반)으로 진화한 흔적

> **설계 변경 이력**: 문서(`docs/`)는 면적당 밀도(cm²) 기준이지만, 실제 코드는 영상에서 추정한 **L당 밀도(per_liter)** 기준으로 동작. 마이그레이션 구문이 이 전환을 반영.

## 7. soft delete + 복구 메커니즘

```
삭제: culture_boxes.deleted_at = now()
      + trash_events(action='deleted', payload={box, measurements})  ← 한 트랜잭션
복구: culture_boxes.deleted_at = NULL
      + trash_events(action='restored')
```
물리 삭제하지 않으므로 측정 이력이 그대로 보존되고 복구 시 즉시 재사용 가능.

---

## 8. DB 특징 정리

1. **정규화 분리** — 변경 주기/용도별 테이블 분리
2. **JSONB 병행** — 정형 컬럼 + 모델 원본(`result_summary`/`*_quality`) 동시 보관
3. **CASCADE 삭제** — measurement 삭제 시 상세 결과 자동 정리
4. **soft delete** — `deleted_at` + `trash_events` 이중 추적
5. **멱등 마이그레이션** — 스키마 진화 안전성 확보
